記錄目前已知限制、評估過但暫緩的方向、以及後續可能的工作，避免未來（不論是你或是 AI）重新踩一次已經踩過的坑。
具體可執行的待辦清單見 [TODO.md](./TODO.md)；這裡偏「方向與背景」，TODO 偏「下一步做什麼」。

## 已知限制（目前狀態，非 bug，但要知道）

- **電力供需即時報表拿不到真即時**：主要來源 `www.taipower.com.tw/.../sys_dem_sup.csv` 從 GitHub Actions
  執行會回 403，判斷是 WAF 封鎖雲端 CI 的 IP 網段，換瀏覽器標頭/Referer 都無法解決。目前落到政府開放資料備援
  （資料集 19995），但該備援雖名為「每日」，實測落後約 6 週，不是「昨天」而已。前端已誠實標示「非即時，截至
  YYYY-MM-DD」，不會跟即時風力出力並列誤導。
- **37331 回填是季度檔，不是每日檔**：原本期待能補足近 7 天趨勢窗的缺口，但實測資料落後約 4–5 個月，
  永遠補不到近期缺口。目前定位改為「長期存檔」（`wind_history_archive.json` / `wind_archive_daily.json`），
  供月/季/年趨勢分析用；近期趨勢缺口仍只能靠 scraper 本身的取樣密度。
- **離岸風場開發歷程只深化了 15 座**：陸域個別風場與「其它台電自有／其它購電風力」這類彙總項目，
  本來就沒有可查證的公開個別詳細紀錄（環評、動工日期等），維持原本的概略一行描述，沒有強行編造。
- **風速僅陸上測站參考值**：中央氣象署測站在陸上/沿海，外海風場（尤其離岸 35–60 km 者）拿不到真正輪轂高度
  （100m+）實測風速，只能用最近測站當參考值，前端已明確標示。曾評估改用中央氣象署浮標（外海海象觀測），
  但浮標數量少、多數離風場位置遠，效益不高，暫緩（見下方「評估過但暫緩」）。
- **時效敏感的風場進度需要定期回頭核對**：以下幾筆資料在寫入時是「最新已知進度」，但計畫本身仍在推進，
  時間點會變動，需要之後有人（或排程）回頭核實：
  - 台電離岸二期：規劃完工已從「2026」更新為「2027」，仍在施工中
  - 海龍（龍B風）：官方最新報導顯示全案商轉可能已由 2026 延至 2027
  - 沃四風／沃南風：屬同一 920 MW 計畫（大彰化 2b&4），規劃 2026 年第三季全數商轉，屆時應核實是否如期
- **git 歷史會持續累積**：`scrape.yml` 每 15 分鐘 commit 一次，一年下來是數萬筆。功能無礙，但 repo 體積會變大；
  `wind_history_archive.json` 也已經 1.8MB 且會隨每週回填持續成長。

## 已評估過、暫緩的方向（附理由，避免重複研究）

- **能源署月/年統計 API**（[ea01.moeaea.gov.tw](https://ea01.moeaea.gov.tw/a0303/02/database/api/)）：
  粒度太粗（月資料），跟現有的即時＋90天日資料重疊度高，加了講不出新故事，暫緩。
- **中央氣象署浮標資料**（外海海象觀測，非陸上測站）：理論上能改善離岸風場的風速參考值準確度，
  但浮標數量少，多數不在離岸風場附近，能改善的風場有限，投入產出比不高，暫緩。
- **環境部離岸風場生態監測資料**（鳥類/白海豚實測監測報告）：若能取代「資訊」頁面目前引用的國際研究數據，
  換成本土實測會更有說服力，但這類資料多為 PDF 報告，非結構化、非機器可讀格式，暫緩，等有結構化資料源再評估。
- **電力供需即時報表的 WAF 封鎖**：曾嘗試改用真實瀏覽器 User-Agent + Referer 重試，403 依舊，判斷是
  IP 網段層級的封鎖，不是標頭問題。若要解決需要換執行環境（見 TODO），非單純程式碼修正能解決。

## 後續可能的方向（構想，尚未排入 TODO）

- **電力供需的長期趨勢**：目前風力已有「即時 → 7天 → 90天長期趨勢」三層，電力供需目前只有「即時（或每日備援）」
  一層。若之後想比照風力做法，可以幫 `grid_status` 也做一個滾動歷史 + 長期存檔 + 前端趨勢圖，
  呈現「備轉容量率」隨時間的變化，甚至疊加風電貢獻做相關性分析。
- **多能源別擴展**：目前網站聚焦風電；genary 資料本身其實含水力、太陽能、火力、核能全機組資料
  （scraper 已經在抓，只是只取風力列）。若想做「全能源別即時監測」，資料源已經到位，是前端呈現的問題。
  這是範疇擴張的重大決定，先跟使用者確認方向再動工，不要自作主張擴大成「全電力監測網站」。
- **陸域風場開發歷程深化**：若未來陸域風場也有更多可查證的公開開發時程資訊（例如地方政府新聞稿、
  環評追蹤系統），可以比照離岸風場的做法（見本次深化 15 座離岸風場的 PR）逐步補齊。

## 部署穩定性

- Public repo 連續 60 天無活動會被 GitHub 自動停用排程 workflow；目前靠每次功能開發間接維持活動，
  沒有專門的 keepalive 機制。若未來開發頻率降低，建議加一支簡單的月排程 workflow 防止被停用，
  或改用 Cloudflare Worker Cron（見 `DEPLOY.md` 方案 B，排程更準時、無 commit 累積問題）。
